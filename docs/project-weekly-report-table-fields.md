# 项目周报看板数据表字段设计

本文档用于指导“项目周报看板”相关数据表建设。目标是让 PM 不需要手工整理周报，而是由系统从多张业务表聚合生成项目周报看板和 AI 管理层总结。

## 总体原则

- 飞书侧当前先建设 6 张业务表：项目状态/风险、任务/协作事项、缺陷、测试概况、资源投入、交付里程碑。
- 第 7 张 `ProjectWeeklySnapshot` 是系统数据库快照表，不需要在飞书里创建。
- 所有飞书业务表都建议保留 `项目ID` 和 `项目名称`，系统匹配优先级为：`项目ID > 项目名称 > 项目别名`。
- PM 只主责维护“项目状态/风险表”；缺陷、测试、资源、里程碑由对应角色维护。
- 成本/预算字段不进入项目周报。
- `测试执行表` 暂不启用；当前用 `测试概况表` 提供测试执行率、通过率、阻塞数等周报指标。
- 单选/多选字段建议尽量使用本文档给出的固定选项，后续系统做统计、筛选、AI 分析时更稳定。
- `项目名称`、`所属项目`、`部门`、`模块` 这类字段可以做成单选，但选项通常随组织和项目变化，不建议在本文档里固定死。

## 1. 项目状态 / 风险表

维护角色：PM 主责，项目1号位协同。  
用途：周报总览、风险灯、周进展、交付范围、管理层支持事项。

| 字段名 | 飞书字段类型 | 必填 | 维护角色 | 用途 |
| --- | --- | --- | --- | --- |
| 序号 | 数字 | 否 | 系统/PM | 排序展示 |
| 项目ID | 文本 | 建议必填 | PM | 系统项目匹配主键 |
| 项目名称 | 文本 | 必填 | PM | 展示与兜底匹配 |
| 项目阶段 | 单选 | 否 | PM | 展示、阶段分析 |
| 交付状态 | 单选 | 否 | PM/项目1号位 | 交付健康判断 |
| 项目1号位 | 文本/人员 | 否 | 项目1号位/PM | 责任展示 |
| PM | 文本/人员 | 必填 | PM/项目主管 | PM 过滤与责任展示 |
| 风险情况 | 单选 | 必填 | PM | 红灯/黄灯/绿灯/未填 |
| 风险趋势 | 单选 | 否 | PM | 上升/持平/下降 |
| 主要风险类型 | 多选 | 否 | PM | 风险分类统计 |
| 关键风险摘要 | 多行文本 | 否 | PM | AI 风险总结 |
| 风险影响范围 | 多行文本 | 否 | PM | 风险影响分析 |
| 交付范围 | 多行文本 | 建议必填 | PM/项目1号位 | 周报交付范围 |
| 近期重点演示 | 单选 | 否 | PM | 近期演示/非演示/待确认 |
| 周进展（PM） | 多行文本 | 建议必填 | PM | 周报进展正文 |
| Daily风险求助（PM） | 多行文本 | 否 | PM | 风险求助与升级 |
| 最紧急的缺人情况（PM视角） | 多行文本 | 否 | PM | 资源缺口风险 |
| 风险解决情况 | 多行文本 | 否 | PM | 风险闭环进展 |
| 下一步动作 | 多行文本 | 否 | PM | 行动闭环 |
| 动作负责人 | 文本/人员 | 否 | PM | 行动责任人 |
| 动作截止时间 | 日期 | 否 | PM | 行动时限 |
| 是否需管理层支持 | 单选 | 否 | PM | 是/否 |
| 需支持事项 | 多行文本 | 否 | PM | 管理层推动事项 |
| 质量状态与GAP | 多行文本 | 否 | QA/PM | 质量风险摘要 |
| 质量等级 | 单选 | 否 | QA/PM | 高/中/低/正常 |
| 更新时间 | 日期时间 | 否 | PM | 数据新鲜度 |
| 更新人 | 文本/人员 | 否 | PM | 更新责任 |

建议单选项：

- `项目阶段`：需求澄清、开发联调、测试验证、演示准备、交付验收、已完成、暂停
- `交付状态`：正常、关注、延期风险、已延期、已完成
- `风险情况`：红灯、黄灯、绿灯、未填
- `风险趋势`：上升、持平、下降
- `近期重点演示`：近期演示、非演示、待确认
- `是否需管理层支持`：是、否
- `质量等级`：高风险、中风险、低风险、正常、待确认

选项说明：

| 字段名 | 字段类型 | 建议选项 | 备注 |
| --- | --- | --- | --- |
| 项目阶段 | 单选 | 需求澄清、开发联调、测试验证、演示准备、交付验收、已完成、暂停 | 用于阶段分布和周报阶段说明 |
| 交付状态 | 单选 | 正常、关注、延期风险、已延期、已完成 | 用于交付健康判断 |
| 风险情况 | 单选 | 红灯、黄灯、绿灯、未填 | 建议严格固定，系统按该字段统计风险灯 |
| 风险趋势 | 单选 | 上升、持平、下降 | 用于判断风险是否恶化 |
| 主要风险类型 | 多选 | 需求风险、进度风险、质量风险、资源风险、客户风险、技术风险、依赖风险、交付风险、数据风险、环境风险、其他 | 可多选，用于风险分类统计 |
| 近期重点演示 | 单选 | 近期演示、非演示、待确认 | 用于管理层关注筛选 |
| 是否需管理层支持 | 单选 | 是、否 | 用于升级事项识别 |
| 质量等级 | 单选 | 高风险、中风险、低风险、正常、待确认 | 用于质量维度健康判断 |

| 更新时间 | 日期时间 | 否 | 任务负责人 | 数据新鲜度 |
| 更新人 | 文本/人员 | 否 | 任务负责人 | 更新责任 |

建议单选项：

- `任务类型`：需求、开发、测试、联调、演示、交付、问题处理、会议行动项、其他
- `状态`：待办、进行中、已完成、已关闭、阻塞、暂停、取消
- `优先级`：高、中、低
- `是否阻塞`：是、否
- `风险等级`：高、中、低、无
- `里程碑`：是、否

选项说明：

| 字段名 | 字段类型 | 建议选项 | 备注 |
| --- | --- | --- | --- |
| 所属项目 | 单选/文本 | 按当前组织项目列表维护 | 建议选项值与系统项目名称一致 |
| 任务类型 | 单选 | 需求、开发、测试、联调、演示、交付、问题处理、会议行动项、其他 | 用于任务分类统计 |

| 优先级 | 单选 | 高、中、低 | 用于风险排序 |

| 里程碑 | 单选 | 是、否 | 标记是否为关键节点任务 |

## 3. 缺陷表

维护角色：QA / 测试负责人主责。  
用途：缺陷新增/关闭/未关闭、严重缺陷、缺陷负责人排行、质量风险分析。

| 字段名 | 飞书字段类型 | 必填 | 维护角色 | 用途 |
| --- | --- | --- | --- | --- |
| 项目ID | 文本 | 建议必填 | QA | 项目匹配 |
| 所属项目 | 文本/单选 | 必填 | QA | 展示与兜底匹配 |
| 缺陷ID | 文本 | 建议必填 | QA/系统 | 去重 |
| request_id | 文本 | 否 | QA | 外部请求追踪 |

| 指向人 | 文本/人员 | 否 | QA | 问题指向 |

建议单选项：

- `问题类型`：工程实现类、模型能力类、Prompt / 策略类、数据问题类、环境 / 版本类、需求 / 规格类、依赖系统类、测试用例类、体验优化类、非缺陷 / 待确认

- `严重等级`：P0、P1、P2、P3、P4
- `处理优先级`：立即处理、今日处理、本周处理、排期处理、暂缓处理、观察跟进
- `状态`：新建、处理中、已修复、已验证、已关闭、已驳回
- `修复状态`：未开始、修复中、待验证、验证通过、验证失败、无需修复

| 问题类型 | 单选 | 工程实现类、模型能力类、Prompt / 策略类、数据问题类、环境 / 版本类、需求 / 规格类、依赖系统类、测试用例类、体验优化类、非缺陷 / 待确认 | 核心分类字段，用于决定处理路径；不参与现有周报统计 |
| 处理路径 | 单选 | 开发修复、模型优化、Prompt / 策略调整、数据修正、环境处理、需求澄清、依赖方推动、测试口径调整、暂缓 / 观察、关闭 / 非问题 | 用于说明下一步处理方式 |
| 依赖方 | 单选/文本 | 模型、端侧、云端、产品、QA、客户、第三方、火山、其他 | 可按项目实际依赖方补充 |
| 是否阻塞主流程 | 单选 | 是、否、待确认 | 用于日会筛选重点问题 |

| 处理结论 | 单选 | 已修复、已规避、已转需求、已转依赖方、已调整测试口径、非问题关闭、暂缓处理、待继续跟进 | 用于缺陷闭环复盘 |
| 模块 | 单选/文本 | 产品需求、端侧、云端、算法、语音、导航、HMI、数据、测试环境、交付集成、其他 | 模块通常按项目变化，可按项目实际补充 |

## 8. 选项字典与填写说明

本章节用于解释所有单选/多选字段的选项含义和填写口径。飞书建表时建议把这些说明同步到字段描述里，方便维护人理解。

### 8.1 项目状态 / 风险表

#### 项目阶段

| 选项 | 含义 |
| --- | --- |
| 需求澄清 | 项目还在确认需求、范围、验收标准或客户输入 |
| 开发联调 | 已进入开发、接口联调、端云联调或功能集成阶段 |
| 测试验证 | 已进入 SIT、UAT、回归、演示前验证等测试阶段 |
| 演示准备 | 近期有演示、评审、客户展示，需要重点保障 |
| 交付验收 | 已进入交付、验收、上线前确认或客户签收阶段 |
| 已完成 | 本阶段项目已完成，主要动作已闭环 |
| 暂停 | 项目因客户、资源、商务、范围等原因暂时停止推进 |

怎么填：

- 当前主要工作在哪个阶段，就填哪个阶段。
- 如果同时处于多个阶段，优先填对管理层风险判断影响最大的阶段，例如“演示准备”优先于“开发联调”。

#### 交付状态

| 选项 | 含义 |
| --- | --- |
| 正常 | 当前交付节奏符合计划，没有明显延期信号 |
| 关注 | 有偏差或不确定性，但暂未影响核心交付 |
| 延期风险 | 存在较明确延期可能，需要提前协调资源或范围 |
| 已延期 | 已经晚于计划/承诺时间 |
| 已完成 | 对应交付范围已完成并得到确认 |

怎么填：

- `正常`：计划、资源、质量基本可控。
- `关注`：有问题但还能通过常规项目管理手段解决。
- `延期风险`：如果不干预，大概率会延期。
- `已延期`：已经错过计划交付或承诺交付时间。

#### 风险情况

| 选项 | 含义 |
| --- | --- |
| 红灯 | 存在重大风险，可能影响演示、交付、上线或客户承诺 |
| 黄灯 | 存在风险或不确定性，需要持续跟进 |
| 绿灯 | 当前整体可控，暂无明显风险 |
| 未填 | 暂无有效风险判断 |

怎么填：

- `红灯`：核心链路失败、关键资源缺失、客户承诺受影响、P0/P1 质量风险明显。
- `黄灯`：有延期、质量、资源、依赖问题，但仍有缓冲和解决路径。
- `绿灯`：进度、质量、资源、交付都基本可控。
- `未填`：数据不完整时临时使用，建议尽快补齐。

周报会用它做两件事：

- 统计红/黄/绿灯项目数量。
- 对红灯项目优先进入管理层风险摘要。

#### 风险趋势

| 选项 | 含义 |
| --- | --- |
| 上升 | 风险比上周更严重，影响范围或发生概率变大 |
| 持平 | 风险没有明显变化 |
| 下降 | 风险正在缓解，影响或概率下降 |

怎么填：

- 对比上一周或上一轮状态填写。
- 如果本周新增关键风险，通常填 `上升`。
- 如果风险动作已推进、缺口变小，填 `下降`。

#### 主要风险类型

| 选项 | 含义 |
| --- | --- |
| 需求风险 | 需求不清、范围频繁变化、验收标准不明确 |
| 进度风险 | 排期紧、关键任务延期、里程碑可能错过 |
| 质量风险 | 缺陷多、测试通过率低、核心链路不稳定 |
| 资源风险 | 人力不足、关键人冲突、资源未确认 |
| 客户风险 | 客户输入不足、客户决策延迟、客户预期变化 |
| 技术风险 | 技术方案不确定、性能/稳定性/架构风险 |
| 依赖风险 | 外部系统、供应商、其他团队依赖未闭环 |
| 交付风险 | 交付范围、版本、环境、验收存在不确定性 |
| 数据风险 | 测试数据、训练数据、业务数据不完整或不可用 |
| 环境风险 | 测试环境、车机环境、云环境、账号权限不可用 |
| 其他 | 不属于以上类型，但需要在风险摘要中说明 |

怎么填：

- 可多选，只选择主要风险，不要把所有可能性都选上。
- 如果选了 `其他`，必须在 `关键风险摘要` 里写清楚。

#### 近期重点演示

| 选项 | 含义 |
| --- | --- |
| 近期演示 | 近期需要客户演示、管理层演示、样板演示或重大评审 |
| 非演示 | 当前没有近期演示压力 |
| 待确认 | 是否演示还未明确 |

怎么填：

- 一般以未来 1-2 周是否有重要演示为准。
- 只要演示会影响资源、质量或交付优先级，就填 `近期演示`。

#### 是否需管理层支持

| 选项 | 含义 |
| --- | --- |
| 是 | 当前问题需要管理层协调资源、客户、范围、优先级或跨部门决策 |
| 否 | 当前团队可在项目组内自行闭环 |

怎么填：

- 需要跨部门资源、客户拍板、范围调整、优先级协调时填 `是`。
- 填 `是` 时，`需支持事项` 必须写具体动作。

#### 质量等级

| 选项 | 含义 |
| --- | --- |
| 高风险 | 质量问题可能影响演示、交付或上线 |
| 中风险 | 有质量问题，需要重点跟进，但尚未阻塞主流程 |
| 低风险 | 有少量问题，整体可控 |
| 正常 | 当前质量状态基本可控 |
| 待确认 | 测试数据不足，暂时无法判断 |

怎么填：

- 结合缺陷严重等级、测试通过率、核心链路稳定性判断。
- 如果 P0/P1 未闭环或核心链路失败，优先填 `高风险`。

### 8.2 任务 / 协作事项表

#### 任务类型

| 选项 | 含义 |
| --- | --- |
| 需求 | 需求澄清、需求评审、需求变更、验收标准确认 |
| 开发 | 编码、接口开发、功能实现、技术方案落地 |
| 测试 | 测试准备、测试执行、缺陷复测、回归验证 |
| 联调 | 端云联调、系统联调、接口联调、集成验证 |
| 演示 | 演示准备、演示脚本、演示环境、演示材料 |
| 交付 | 版本发布、交付包准备、客户验收、上线支持 |
| 问题处理 | 缺陷、阻塞、风险、客户反馈的处理动作 |
| 会议行动项 | 会议纪要里明确的待办动作 |
| 其他 | 不属于以上类型的事项 |

怎么填：

- 按任务的主要目的填写，不按执行人的部门填写。
- 如果是会议产生的开发任务，优先按实际内容填 `开发`，会议跟踪事项才填 `会议行动项`。

#### 状态

| 选项 | 含义 |
| --- | --- |
| 待办 | 尚未开始 |
| 进行中 | 已开始但未完成 |
| 已完成 | 任务已完成，结果可用 |
| 已关闭 | 任务已确认闭环，不再跟踪 |
| 阻塞 | 因依赖、资源、环境、客户输入等原因无法继续 |
| 暂停 | 暂时不推进，但后续可能恢复 |
| 取消 | 任务已取消，不再执行 |

怎么填：

- `已完成/已关闭` 会计入完成。
- `阻塞` 必须填写 `阻塞原因` 和下一步动作。
- `暂停/取消` 不应当用来隐藏延期任务，必须有明确原因。

#### 优先级

| 选项 | 含义 |
| --- | --- |
| 高 | 影响关键路径、客户承诺、演示、交付或重大风险闭环 |
| 中 | 重要但不在最关键路径上 |
| 低 | 不影响主流程，可延后处理 |

怎么填：

- 影响本周交付、演示或风险闭环的任务，优先级应为 `高`。
- 不要把所有任务都填高，否则周报无法排序。

#### 是否阻塞

| 选项 | 含义 |
| --- | --- |
| 是 | 当前任务无法继续推进，需要外部输入、资源或决策 |
| 否 | 当前任务可以继续推进 |

怎么填：

- 填 `是` 时，必须填写 `阻塞原因`。
- 如果只是进度慢但仍能推进，不要填 `是`，应在风险等级或进度里体现。

#### 风险等级

| 选项 | 含义 |
| --- | --- |
| 高 | 可能影响项目关键路径或客户承诺 |
| 中 | 有一定影响，需要跟进 |
| 低 | 影响较小，可常规处理 |
| 无 | 当前无明显风险 |

怎么填：

- 结合任务重要性、截止时间、阻塞状态判断。
- 已阻塞且影响关键路径的任务，应填 `高`。

#### 里程碑

| 选项 | 含义 |
| --- | --- |
| 是 | 该任务是关键里程碑或里程碑前置任务 |
| 否 | 普通任务 |

怎么填：

- 影响阶段切换、演示、交付、验收的任务填 `是`。

### 8.3 缺陷表

#### 字段关系说明

- `模块` 表示缺陷发生在哪个业务/技术模块，例如记忆、导航、端侧、云端。
- `问题类型` 表示缺陷成因或处理路径，例如工程实现类、模型能力类、依赖系统类。
- `严重等级` 表示影响程度，P0/P1 会进入现有周报严重缺陷统计。
- `处理优先级` 只用于人工排队，不参与现有周报看板展示、分析和计算。

#### 问题类型

| 选项 | 含义 | 典型处理方式 |
| --- | --- | --- |
| 工程实现类 | 代码逻辑、接口调用、状态处理、端云实现问题 | 分配开发修复 |
| 模型能力类 | 模型理解、推理、识别、泛化能力不足 | 交模型/算法团队评估 |
| Prompt / 策略类 | 提示词、规则、策略、工具调用编排问题 | 调整 Prompt、策略或工具链 |
| 数据问题类 | 测试数据、业务数据、训练数据、配置数据问题 | 补数据、修数据、清洗数据 |
| 环境 / 版本类 | 环境、设备、版本、网络、账号导致的问题 | 更换环境、升级版本、补账号 |
| 需求 / 规格类 | 需求不清、验收标准冲突、规则未定义 | PM/产品澄清需求 |
| 依赖系统类 | 依赖第三方、客户系统、端侧、火山等外部能力 | 推动依赖方处理 |
| 测试用例类 | 用例设计、预期结果、测试口径问题 | QA 修正用例或测试口径 |
| 体验优化类 | 不影响功能正确性，但体验不佳 | 排期优化 |
| 非缺陷 / 待确认 | 暂无法确认是缺陷，或实际不是问题 | 等待复现或关闭 |

怎么填：

- QA 创建缺陷时先做初判，无法判断时填 `非缺陷 / 待确认` 或在 `分类备注` 里说明。
- 日会中优先看 `问题类型 + 是否阻塞主流程 + 严重等级`，快速区分要开发修、模型看、客户协调还是需求澄清。
- 不要用 `模块` 代替 `问题类型`：例如导航模块里的问题也可能是工程实现类、模型能力类或依赖系统类。

#### 处理路径

| 选项 | 含义 |
| --- | --- |
| 开发修复 | 明确是工程问题，直接进入开发修复 |
| 模型优化 | 需要模型、算法、识别、推理能力优化 |
| Prompt / 策略调整 | 通过 Prompt、规则、工具链策略调整解决 |
| 数据修正 | 通过补充、修正、清洗数据解决 |
| 环境处理 | 通过环境、版本、账号、设备处理解决 |
| 需求澄清 | 需要 PM/产品/客户确认规则或验收标准 |
| 依赖方推动 | 需要外部系统、客户或第三方团队处理 |
| 测试口径调整 | 测试预期、用例或判定标准需要调整 |
| 暂缓 / 观察 | 暂时无法处理或优先级不足，继续观察 |
| 关闭 / 非问题 | 确认不是缺陷或无需修复 |

怎么填：

- `问题类型` 是分类，`处理路径` 是动作，两者可以不同。
- 例如问题类型为 `依赖系统类`，处理路径通常为 `依赖方推动`；问题类型为 `需求 / 规格类`，处理路径通常为 `需求澄清`。

#### 依赖方

| 选项 | 含义 |
| --- | --- |
| 模型 | 依赖模型、算法、识别、推理能力 |
| 端侧 | 依赖车机端、客户端或本地执行 |
| 云端 | 依赖服务端、接口、云服务或数据服务 |
| 产品 | 依赖产品规则、需求、验收标准 |
| QA | 依赖测试用例、测试数据、测试口径 |
| 客户 | 依赖客户确认、客户环境、客户接口或账号 |
| 第三方 | 依赖外部供应商或第三方服务 |
| 火山 | 依赖火山相关能力或平台支持 |
| 其他 | 不属于以上依赖方 |

怎么填：

- 没有明确依赖方时可以留空。
- 如果依赖方会影响闭环节奏，建议同时在 `分类备注` 里写清楚需要对方提供什么。

#### 是否阻塞主流程

| 选项 | 含义 |
| --- | --- |
| 是 | 阻塞主链路、演示、交付或核心验收 |
| 否 | 不阻塞主流程 |
| 待确认 | 影响范围还未确认 |

怎么填：

- 影响近期演示、交付、主链路测试、P0/P1 清零目标时填 `是`。
- 只是体验优化、边界问题或可绕行问题通常填 `否`。

#### 根因判断

| 选项 | 含义 |
| --- | --- |
| 代码实现 | 代码逻辑、状态处理、异常处理、执行链路问题 |
| 接口 / 协议 | 接口参数、协议字段、上下游调用不一致 |
| 模型能力 | 模型理解、识别、推理、泛化能力不足 |
| Prompt / 策略 | 提示词、规则、策略、工具调用编排问题 |
| 数据 | 测试数据、业务数据、训练数据、配置数据问题 |
| 环境 / 版本 | 环境、版本、设备、网络、账号问题 |
| 需求不清 | 需求规则、验收标准、边界条件未定义 |
| 依赖方 | 外部系统、客户、第三方能力未满足 |
| 测试口径 | 测试用例、预期结果、判定标准问题 |
| 待定位 | 根因尚未确认 |

#### 处理结论

| 选项 | 含义 |
| --- | --- |
| 已修复 | 通过修复代码、配置、策略等完成闭环 |
| 已规避 | 通过绕行方案或临时处理规避风险 |
| 已转需求 | 确认为需求新增或规则补充 |
| 已转依赖方 | 已转给外部依赖方继续处理 |
| 已调整测试口径 | 确认为测试预期或用例口径问题 |
| 非问题关闭 | 确认不是缺陷或无需修复 |
| 暂缓处理 | 当前不处理，后续排期或观察 |
| 待继续跟进 | 仍未完全闭环，需要继续推进 |

#### 处理优先级

| 选项 | 含义 |
| --- | --- |
| 立即处理 | 当前阻塞演示、交付、主流程验证，需要马上跟进 |
| 今日处理 | 当天需要闭环或给出明确进展 |
| 本周处理 | 本周内处理，不影响当天主流程 |
| 排期处理 | 进入后续版本或迭代计划 |
| 暂缓处理 | 当前不处理，等待复现、客户确认或版本窗口 |
| 观察跟进 | 暂无明确处理动作，持续观察或等待更多数据 |

怎么填：

- 阻塞演示/交付填 `立即处理`。
- 影响当天测试推进填 `今日处理`。
- 普通缺陷但本周要清填 `本周处理`。
- 非本周版本范围填 `排期处理`。

#### 模块

| 选项 | 含义 |
| --- | --- |
| 产品需求 | 需求定义、交互规则、验收标准相关问题 |
| 端侧 | 车机端、移动端、本地客户端相关问题 |
| 云端 | 服务端、接口、数据库、云服务相关问题 |
| 算法 | 模型、推荐、识别、推理、算法策略相关问题 |
| 语音 | 唤醒、识别、语义、播报、多轮对话相关问题 |
| 导航 | 地图、路线、定位、ETA、导航播报相关问题 |
| HMI | UI、交互、视觉、动效、布局相关问题 |
| 数据 | 测试数据、业务数据、训练数据、同步数据相关问题 |
| 测试环境 | 环境、账号、设备、版本、网络导致的问题 |
| 交付集成 | 版本集成、交付包、客户环境、验收相关问题 |
| 其他 | 不属于以上模块的问题 |

怎么填：

- 按缺陷最直接归属模块填写。
- 如果跨多个模块，优先填主责模块，其他影响写到问题描述或上下文。

#### 严重等级

| 选项 | 含义 |
| --- | --- |
| P0 | 致命问题，核心功能不可用、系统崩溃、演示/交付无法进行 |
| P1 | 严重问题，核心链路失败或重要功能不可用 |
| P2 | 一般问题，影响部分功能但有绕行方案 |
| P3 | 轻微问题，体验、显示、边界场景问题 |
| P4 | 建议类、优化类或极低影响问题 |

怎么填：

- P0/P1 会进入周报严重缺陷统计。
- 如果问题影响演示、交付或客户核心验收，至少按 P1 评估。

#### 状态

| 选项 | 含义 |
| --- | --- |
| 新建 | 缺陷刚创建，尚未开始处理 |
| 处理中 | 开发或责任人正在处理 |
| 已修复 | 已提交修复，等待验证 |
| 已验证 | QA 已验证通过 |
| 已关闭 | 缺陷已完成闭环 |
| 已驳回 | 确认不是缺陷，或无需修复 |

怎么填：

- `新建/处理中/已修复` 会计入未关闭缺陷。
- `已验证/已关闭/已驳回` 会计入关闭或不再处理。

#### 修复状态

| 选项 | 含义 |
| --- | --- |
| 未开始 | 已分派但尚未开始修复 |
| 修复中 | 正在修复 |
| 待验证 | 修复完成，等待 QA 验证 |
| 验证通过 | QA 验证通过 |
| 验证失败 | 修复后仍未通过，需要继续处理 |
| 无需修复 | 确认不是问题或无需改动 |

怎么填：

- 用于比 `状态` 更细地描述修复过程。
- 如果 `状态=已修复`，通常 `修复状态=待验证`。

### 8.4 测试概况表

#### 测试轮次

| 选项 | 含义 |
| --- | --- |
| 冒烟测试 | 快速验证版本主链路是否可测、是否具备进入下一轮测试条件 |
| SIT | 系统集成测试，验证多个模块集成后的功能和链路 |
| UAT | 用户验收测试，验证是否满足客户/业务验收标准 |
| 回归测试 | 验证修复后是否引入新问题，确认历史功能稳定 |
| 专项测试 | 针对性能、稳定性、兼容性、安全性等专项验证 |
| 演示前验证 | 演示前对关键路径、演示脚本、环境和数据进行验证 |
| 其他 | 不属于以上轮次的测试活动 |

怎么填：

- 按本行统计数据对应的测试活动填写。
- 一次测试活动跨多个模块时，`模块` 多选即可，不需要拆成多行。

#### 模块

| 选项 | 含义 |
| --- | --- |
| 产品需求 | 本轮覆盖需求、验收标准、业务规则相关内容 |
| 端侧 | 本轮覆盖车机端、移动端或客户端能力 |
| 云端 | 本轮覆盖服务端、接口、数据服务能力 |
| 算法 | 本轮覆盖算法、模型、策略相关能力 |
| 语音 | 本轮覆盖语音唤醒、识别、语义、播报、多轮对话 |
| 导航 | 本轮覆盖地图、定位、路线规划、导航播报 |
| HMI | 本轮覆盖 UI、交互、视觉和动效 |
| 数据 | 本轮覆盖数据同步、数据准确性、数据准备 |
| 测试环境 | 本轮主要验证环境、设备、账号、网络或版本可用性 |
| 交付集成 | 本轮覆盖交付包、集成版本、客户环境验收 |
| 其他 | 本轮覆盖其他模块 |

怎么填：

- 多选本轮实际覆盖的模块。
- 不要把计划覆盖但实际未执行的模块选上，避免覆盖率误判。

#### 测试结论

| 选项 | 含义 |
| --- | --- |
| 通过 | 本轮测试结果整体可接受，可以进入下一阶段 |
| 关注 | 有问题但不阻塞主流程，需要持续跟进 |
| 高风险 | 存在明显质量风险，可能影响演示、交付或上线 |
| 阻塞 | 当前无法继续有效测试或无法推进下一阶段 |
| 待确认 | 数据还不完整，暂时不能下结论 |

怎么填：

- `通过`：通过率较高，没有关键阻塞，P0/P1 缺陷可控。
- `关注`：有失败或缺陷，但不影响主链路推进。
- `高风险`：P0/P1 较多、核心链路失败、通过率明显偏低。
- `阻塞`：环境、版本、数据、依赖等导致测试无法继续。
- `待确认`：刚开始测试，数据不完整，暂时不下判断。

周报会用它做两件事：

- 在质量与测试区域展示 QA 的总体判断。
- 结合通过率、阻塞数、严重缺陷数生成管理层质量结论。

### 8.5 测试执行表（可选）

#### 优先级

| 选项 | 含义 |
| --- | --- |
| 高 | 核心链路、客户验收、演示或交付必须覆盖的重点用例 |
| 中 | 常规功能用例，对质量判断有参考价值 |
| 低 | 边界、体验、非核心路径或低频场景用例 |

怎么填：

- 影响演示、交付、上线、主链路的用例填 `高`。
- 不要把所有用例都填高，否则无法识别重点测试范围。

#### 执行状态

| 选项 | 含义 |
| --- | --- |
| 未执行 | 用例还没有开始执行 |
| 通过 | 已执行，实际结果符合预期 |
| 失败 | 已执行，实际结果不符合预期 |
| 阻塞 | 因环境、版本、数据、依赖等原因无法有效执行 |
| 跳过 | 本轮明确不执行该用例 |

怎么填：

- `通过`：不用填失败原因、阻塞原因。
- `失败`：建议填写 `失败原因`，如已建缺陷则填写 `关联缺陷ID`。
- `阻塞`：必须填写 `阻塞原因`。
- `未执行`：执行人、执行时间可以为空。
- `跳过`：建议在备注里说明跳过原因。

周报会用它做四件事：

- 计算测试执行率。
- 计算测试通过率。
- 统计失败/阻塞用例数。
- 识别模块测试质量风险。

### 8.6 资源投入表

#### 人员ID 与姓名

| 字段 | 含义 |
| --- | --- |
| 姓名 | 必填，作为周报资源投入统计的主字段 |
| 人员ID | 非必填，仅在团队已有统一工号、LDAP、人员编号或飞书人员资源表编号时填写 |

怎么填：

- 优先填写 `姓名`，没有统一人员编号时 `人员ID` 留空即可。
- 如果存在同名人员，再补充 `人员ID` 做区分。
- 周报看板当前按 `姓名` 展示和统计；`人员ID` 只作为辅助识别字段。

#### 投入口径

周报看板按 **周口径** 统计资源投入：

- 一条资源记录只有在 `开始日期 ~ 结束日期` 与当前周报周期有交集时，才计入本周资源投入。
- 如果填写了 `投入人天`，系统会按 `本周交集天数 / 整段投入天数` 折算本周人天。
- 如果没有填写 `投入人天`，系统按 `本周交集天数 × 投入比例 / 100` 计算本周人天。
- `投入比例` 按百分比填写，例如 `100` 表示 100% 投入，`50` 表示 50% 投入。

示例：

- 资源投入周期：6/1 至 6/30
- 投入比例：100
- 周报周期：6/8 至 6/14
- 本周资源投入：7 人天

如果整段 `投入人天` 填了 21，系统会按本周交集 7 天相对整段 30 天折算为约 4.9 人天。

#### 角色

| 选项 | 含义 |
| --- | --- |
| 产品 | 产品经理、需求分析、业务规则负责人 |
| 研发 | 通用研发人员，无法细分端侧/云端时使用 |
| 测试 | QA、测试开发、测试负责人 |
| 算法 | 算法、模型、策略相关人员 |
| 端侧 | 车机端、客户端、移动端研发 |
| 云端 | 后端、云服务、接口、数据服务研发 |
| 项目管理 | PM、项目协调、计划跟踪 |
| 交付 | 交付经理、现场支持、客户验收支持 |
| 其他 | 不属于以上角色 |

怎么填：

- 优先填实际承担的项目角色，不一定等同于组织岗位。
- 如果研发人员明确负责云端，填 `云端`，不要泛填 `研发`。

#### 是否关键资源

| 选项 | 含义 |
| --- | --- |
| 是 | 该人员缺失会明显影响关键路径、演示、交付或质量闭环 |
| 否 | 该人员不是当前关键瓶颈资源 |

怎么填：

- 单点专家、关键模块主责、客户现场关键支持一般填 `是`。
- 普通支援或可替代人员填 `否`。

#### 资源状态

| 选项 | 含义 |
| --- | --- |
| 可用 | 当前投入可满足项目需要 |
| 紧张 | 投入不足或排期紧，但还能协调 |
| 过载 | 已明显超负荷，可能影响质量或进度 |
| 请假 | 当前请假或短期不可用 |
| 离岗 | 已离开项目或长期不可用 |
| 待确认 | 资源是否可用尚未确认 |

怎么填：

- 结合投入比例、项目冲突和实际交付压力填写。
- 同一人员多个项目投入叠加过高时，通常填 `紧张` 或 `过载`。

#### 分配类型

| 选项 | 含义 |
| --- | --- |
| 正式投入 | 已纳入项目计划的稳定投入 |
| 临时支援 | 短期帮助解决问题或阶段性补位 |
| 专项支持 | 针对某个专项问题、演示、交付进行支持 |
| 待确认 | 是否投入、投入周期或比例尚未确认 |

怎么填：

- 长期在项目组内承担任务，填 `正式投入`。
- 临时解决问题，填 `临时支援` 或 `专项支持`。

#### 分配确认状态

| 选项 | 含义 |
| --- | --- |
| 已确认 | 资源负责人和项目方都确认投入 |
| 待确认 | 已提出需求，但资源方尚未确认 |
| 被拒绝 | 资源申请被拒绝或无法满足 |
| 需协调 | 存在冲突，需要管理层或部门负责人协调 |

怎么填：

- 只要投入比例、周期或人员未明确，就不要填 `已确认`。
- `被拒绝/需协调` 会进入资源风险提示。

### 8.7 交付里程碑表

#### 里程碑类型

| 选项 | 含义 |
| --- | --- |
| 需求冻结 | 需求范围、验收标准、版本范围确认 |
| 开发完成 | 开发任务完成，可以进入联调或测试 |
| 联调完成 | 端云、系统、接口联调完成 |
| 测试完成 | 测试计划完成，主要质量问题已闭环 |
| 演示 | 客户演示、管理层演示、评审展示节点 |
| 交付 | 版本、交付包、材料或环境交付节点 |
| 验收 | 客户验收、内部验收、阶段验收节点 |
| 上线 | 生产发布、正式上线或对外启用节点 |

怎么填：

- 选择该里程碑最核心的业务属性。
- 如果是演示前交付包准备，优先看它的管理目标：演示就填 `演示`，交付就填 `交付`。

#### 交付状态

| 选项 | 含义 |
| --- | --- |
| 未开始 | 里程碑相关工作尚未开始 |
| 进行中 | 正在推进，尚未完成 |
| 延期风险 | 有较明确延期可能，但还未实际延期 |
| 已延期 | 已超过计划或承诺日期 |
| 已完成 | 已完成并确认 |
| 暂停 | 当前节点暂时停止推进 |

怎么填：

- 超过 `计划交付日期` 或 `承诺交付日期` 且未完成，填 `已延期`。
- 预计无法按期完成但日期未到，填 `延期风险`。

#### 风险等级

| 选项 | 含义 |
| --- | --- |
| 高 | 可能影响演示、交付、验收或客户承诺 |
| 中 | 有一定影响，需要持续跟进 |
| 低 | 影响较小，可常规处理 |
| 无 | 当前无明显风险 |

怎么填：

- 和里程碑交付影响挂钩，不只看任务难度。
- 如果该节点是客户承诺节点，风险等级要更保守。
