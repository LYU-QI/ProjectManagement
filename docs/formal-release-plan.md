# ProjectLVQI 正式版开发计划

## 1. 目标

本文档定义 `ProjectLVQI` 从当前 `Beta 阶段的内部生产工具` 演进到“正式版”的开发范围、实施顺序、验收标准与阶段计划。

正式版目标不是继续堆叠功能，而是把现有能力做成：

- 数据可信
- 交互一致
- 关键流程稳定
- 权限与审计可追溯
- 发布与运维可控

## 2. 当前阶段判断

当前系统已经具备：

- 项目、需求、任务、进度、成本、风险主链路
- 飞书多维表格读写
- PM 助手、AI 分析、能力模板
- 自动化规则、任务中心、SSE 事件流
- 组织/项目维度权限与审计基础

因此当前阶段判断为：

**Beta 阶段的内部生产工具，具备对外演示能力，但尚未达到正式商用交付标准。**

## 3. 正式版准入标准

正式版至少需要满足以下 6 条：

1. 核心指标口径统一
2. 全局上下文一致
3. 关键流程稳定可回归
4. 权限、审计、配置安全完整
5. UI/交互收敛
6. 发布、监控、回滚可执行

## 4. 差距拆解

### 4.1 数据口径

当前差距：

- 成本、工时、AI、报表、飞书之间的口径还存在分散实现
- 相同概念在不同模块存在重复计算
- AI 回答可能基于不完整上下文生成“看似合理但口径不一致”的结果

正式版要求：

- 每个核心指标只有一个定义
- 后端统一汇总，前端与 AI 不重复发明口径
- 任何关键数字都能追到来源

### 4.2 全局上下文

当前差距：

- 当前组织 / 当前项目 / 页面局部筛选在部分页面语义接近但不完全一致
- 飞书页、AI、自动化、任务中心都依赖上下文，但用户可见性不够统一

正式版要求：

- 所有页面明确显示当前组织、当前项目、数据源范围
- “切项目”和“页面筛选”必须严格区分
- API 层显式接收并返回上下文信息

### 4.3 稳定性与测试

当前差距：

- 关键链路虽然可用，但自动化回归不完整
- 同步失败、重试失败、上下文串页等问题仍需手动排查

正式版要求：

- 关键流程有自动化回归
- 异常有分类、上下文、恢复入口
- 数据同步与后台任务状态可观测

### 4.4 权限与审计

当前差距：

- 平台角色、组织角色、项目访问之间仍偏工程实现导向
- 审计虽有基础，但关键动作的上下文字段还需统一

正式版要求：

- 明确权限矩阵
- 关键操作全量审计
- API Key、飞书配置、AI 配置等敏感信息治理到位

### 4.5 UI/交互

当前差距：

- 部分页面存在强桌面化布局、复杂表格、操作入口不统一
- 大屏自适应与复杂筛选体验还在持续修补

正式版要求：

- 主路径页面交互一致
- 大屏、笔记本、常规桌面体验稳定
- 错误态、空态、加载态统一

### 4.6 运维与发布

当前差距：

- 需要进一步标准化环境、监控、发布和回滚流程

正式版要求：

- staging / production 分层
- 数据迁移、配置变更、回滚路径清晰
- 任务失败、外部依赖异常、关键接口异常可监控

## 5. 实施策略

整体采用“先统一、再稳定、后收口”的路线：

1. 先统一核心口径与上下文
2. 再做关键链路稳定性与自动化回归
3. 再补权限、审计、发布与监控
4. 最后收 UI 与交付文档

原则：

- 少加新功能
- 优先消除重复实现和隐式逻辑
- 所有统计与 AI 口径优先复用统一后端服务

## 6. 开发阶段与计划

### 阶段 1：口径与上下文统一

周期：2 周

目标：

- 建立统一项目指标服务
- 统一 AI / 成本 / 报表的财务口径
- 收口当前组织 / 当前项目 / 当前数据源的上下文模型

开发项：

- 新增共享 `project-metrics` 模块
- 成本页、AI 聊天、AI 周报、进展分析统一复用共享指标
- 页面补统一上下文条
- 飞书、任务中心、AI 页面统一显示“当前项目/当前作用域”

验收标准：

- 成本与工时页、AI 聊天、AI 周报中“实际支出”一致
- “当前项目 / 本项目”不再出现跨项目串答
- 同一项目的预算、偏差率、直接成本、工时成本可在多个页面交叉验证一致

### 阶段 2：关键流程稳定性

周期：2-3 周

目标：

- 将主路径从“可用”升级为“稳定”

开发项：

- 关键链路 E2E：
  - 登录
  - 组织切换
  - 项目切换
  - 需求创建/审核/变更
  - 飞书同步
  - 成本/工时录入
  - PM 助手执行
  - 自动化规则执行
- 统一错误码与错误提示
- 重试与失败恢复增强
- 任务中心补更多失败上下文

验收标准：

- 关键链路具备自动化回归
- 飞书 / 自动化 / PM 助手失败可定位
- 失败后可明确知道是否需要人工介入

### 阶段 3：权限、审计、配置安全

周期：2 周

目标：

- 满足正式交付的治理要求

开发项：

- 输出权限矩阵
- 收紧项目级/组织级访问判定
- 审计日志补齐上下文字段
- 敏感配置隔离与脱敏
- 关键管理动作二次确认与可追溯

验收标准：

- 关键管理操作均可追踪操作者、项目、来源与结果
- 越权访问路径基本消除
- 敏感配置不会在 UI 或日志中裸露

### 阶段 4：UI 收口与发布准备

周期：2 周

目标：

- 提升正式版感知质量

开发项：

- 主路径页面大屏自适应优化
- 统一筛选、分页、表格、详情交互
- 统一空态、错误态、加载态
- 管理员手册、发布说明、上线 checklist

验收标准：

- 主路径页面体验一致
- 大屏/常规桌面环境表现稳定
- 文档足以支撑试运行与正式发布
